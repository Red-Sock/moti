## What is moti

`moti` is a CLI tool for managing Protocol Buffers workflows — a fork of `easy-p`. It wraps `protoc` with declarative dependency management (`moti install`) and code generation (`moti generate`) driven by a `moti.yaml` config file.

## Architecture

The codebase follows a clean layered structure:

- **`main.go`** — wires cobra commands; each command is a `commands.Handler` registered via `addCommands`.
- **`internal/commands/`** — CLI entry points (`install/`, `generate/`). Each command embeds `commands.Env` (injected dependencies: `Console`, `Storage`, `LockFile`, `ModuleConfig`, `MotiConfig`). Business logic lives in `core.go`; the cobra command setup is in `command.go`.
- **`internal/config/`** — reads and parses `moti.yaml` into `config.Config`. `ReadOrDie` is the production path; `Read` returns errors for tests.
- **`internal/models/`** — domain types (`Module`, `Revision`, `RequestedVersion`) and sentinel errors. `Module.Version` is `Omitted` when no `@version` suffix is given.
- **`internal/adapters/`** — concrete implementations behind interfaces:
  - `console/` — shell command execution
  - `storage/` — local cache directory management (install, hash, lookup)
  - `lock_file/` — `moti.lock` read/write
  - `module_config/` — reads a remote repo's own `moti.yaml` to resolve transitive deps
  - `repository/git/` — git clone/fetch/archive of remote proto repos
  - `fs/` — filesystem walker abstraction
- **`internal/mocks/`** — generated by `minimock`; regenerate with `go generate ./...` after any interface change.

### Key data flows

**`moti install`**: reads `moti.yaml` deps → for each module, creates a cache repo dir → git-clones/fetches → reads the remote's `moti.yaml` (transitive deps) → archives proto files into `cache_path/mod/<module>/<version>/` → writes hash + version to `moti.lock`.

**`moti generate`**: reads `moti.yaml` generate rules → for each input (local dir or git repo), resolves the install path via `moti.lock` → assembles `protoc -I ... --<plugin>_out=... <files>` → executes via `Console.RunCmd`.

## Conventions

- Use `go.redsock.ru/rerrors` for all error wrapping: `rerrors.Wrap(err, "context")` or `rerrors.Wrap(err)` with no message when the call site is self-explanatory.
- Use `any` instead of `interface{}`.
- Interfaces are named with an `I` prefix (e.g., `IStorage`); implementations drop the prefix.
- YAML/JSON struct tags must use `snake_case` (enforced by `tagliatelle` linter).
- Prefer commit hashes over pseudo-versions when generating or pinning versions.
- When unsure about deleting code during refactors, mark it `// TODO UNUSED` rather than deleting immediately.

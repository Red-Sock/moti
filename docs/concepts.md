# Key concepts

## The lock file (`moti.lock`)

After `moti install` runs, it writes a `moti.lock` file next to `moti.yaml`. Each line records a module, its resolved version, and a content hash:

```
github.com/googleapis/googleapis  v0.0.0-20260324152955-59d5f2b46924714af627ac29ea6de78641a00835  sha256:abc123...
github.com/vervstack/Matreshka    v1.0.95  sha256:def456...
```

The lock file serves two purposes:

1. **Reproducibility** — `moti generate` reads it to find the exact cache path for each module, so generation always uses the same snapshot regardless of upstream changes.
2. **Idempotence** — `moti install` compares the on-disk content hash against the lock file before deciding whether to re-fetch a module.

**Always commit `moti.lock`** to version control.

## The cache directory

Proto files are stored at:

```
<cache_path>/mod/<module>/<version>/
```

For example:

```
proto_modules/mod/github.com/googleapis/googleapis/v0.0.0-abc123/
proto_modules/mod/github.com/vervstack/Matreshka/v1.0.95/
```

The default `cache_path` is `proto_modules`. Add this directory to `.gitignore` — it is regenerated by `moti install`.

## Proto import root

How protoc resolves imports between `.proto` files depends on what you pass as the `-I` flag — the "import root".

moti sets the `-I` root from the `directory` field in an input. This means your proto files must import each other relative to that root.

### Example: directory as root (`directory: api/grpc`)

```
project/
  api/grpc/
    common.proto
    service.proto
```

With `directory: api/grpc`, moti calls `protoc -I api/grpc ...`. Inside `service.proto`, import common as:

```protobuf
import "common.proto";
```

Generated files land directly in `out` (e.g. `internal/api/server/*.go`).

### Example: repo root as root (`directory: .` or omitted)

With `directory: .`, moti calls `protoc -I . ...`. Inside `service.proto`:

```protobuf
import "api/grpc/common.proto";
```

Generated files land in `out` plus the full package path, unless `paths: source_relative` is set.

## `source_relative` vs `module` option

Two common plugin options control output path layout:

### `paths: source_relative`

Generated files mirror the proto file's directory structure relative to the import root, ignoring the `go_package` option in the proto file. Use this when your `go_package` contains a long import path you do not want reflected in the output directory.

```yaml
plugins:
  - name: go
    out: internal/api/server
    opts:
      paths: source_relative
```

### `module: <prefix>`

Strips a Go module prefix from output paths. Use this when generating client code from external repositories whose `go_package` contains a prefix you want removed.

```yaml
plugins:
  - name: go
    out: internal/api/clients
    opts:
      module: go.vervstack.ru
```

This tells the `go` plugin: "all generated files live in the `go.vervstack.ru` module — strip that prefix and write the remainder under `out`."

## Local replace

During active development you may want to consume a local copy of a proto repository instead of the cached version. Add a `replace` entry:

```yaml
replace:
  - old: github.com/vervstack/Matreshka
    new: ../Matreshka
```

- `moti install` skips fetching the replaced module.
- `moti generate` reads proto files from the local path instead of the cache.
- Relative paths are resolved relative to `moti.yaml`.

Remove the `replace` entry before committing to keep CI reproducible.

---

## Transitive dependencies

When `moti install` fetches a module, it reads that module's own `moti.yaml` (if present) and installs its `deps` as well. This means you rarely need to list transitive proto dependencies explicitly — they are resolved automatically.
